Building a Web3 DApp A Detailed Step-by-Step Guide with Code Samples

Wednesday, July 16, 2025 at 3:29:19 PM GMT+8

Web3 decentralized applications (DApps) combine blockchain-based smart contracts, data indexing, off-chain storage, and user-friendly frontends to create powerful, decentralized experiences. In this comprehensive guide, we’ll walk through the full workflow for building a Web3 DApp, using an e-commerce escrow system as an example. We’ll provide detailed explanations and sample code for each step to make the process interactive and actionable. Whether you’re a beginner or an experienced developer, this guide will help you understand how to build a DApp from scratch.

Step 1: Smart Contract (On-Chain Logic)

The smart contract is the core of your DApp, defining the decentralized logic that runs on a blockchain like Ethereum. For our e-commerce escrow DApp, the contract will handle order creation, payment escrow, and release of funds.

What You’ll Do

- Write a Solidity smart contract to manage escrow logic.

- Deploy it to a testnet (e.g., Sepolia) using Foundry.

- The contract will emit events for key actions (e.g., OrderCreated) and have a public address (e.g., 0x...).

Sample Code: EcommerceEscrow.sol

Below is a simple Solidity contract for an e-commerce escrow system. It allows buyers to create orders, holds funds in escrow, and releases them to the seller upon confirmation.

// SPDX-License-Identifier: MIT pragma solidity ^0.8.20;

contract EcommerceEscrow { // Struct to store order details struct Order { uint256 id; address buyer; address seller; uint256 amount; bool isReleased; bool isDisputed; }

// Mapping to store orders
mapping(uint256 => Order) public orders;
uint256 public orderCount;

// Events for indexing
event OrderCreated(uint256 indexed orderId, address indexed buyer, address indexed seller, uint256 amount);
event FundsReleased(uint256 indexed orderId, address indexed seller);
event DisputeRaised(uint256 indexed orderId);

// Create a new order
function createOrder(address _seller) external payable {
    require(msg.value > 0, "Payment must be greater than 0");
    orderCount++;
    orders[orderCount] = Order(orderCount, msg.sender, _seller, msg.value, false, false);
    emit OrderCreated(orderCount, msg.sender, _seller, msg.value);
}

// Release funds to seller
function releaseFunds(uint256 _orderId) external {
    Order storage order = orders[_orderId];
    require(msg.sender == order.buyer, "Only buyer can release funds");
    require(!order.isReleased, "Funds already released");
    require(!order.isDisputed, "Order is disputed");

    order.isReleased = true;
    payable(order.seller).transfer(order.amount);
    emit FundsReleased(_orderId, order.seller);
}

// Raise a dispute
function raiseDispute(uint256 _orderId) external {
    Order storage order = orders[_orderId];
    require(msg.sender == order.buyer || msg.sender == order.seller, "Not authorized");
    require(!order.isReleased, "Funds already released");
    order.isDisputed = true;
    emit DisputeRaised(_orderId);
}

Deployment with Foundry

Install Foundry: Follow the Foundry installation guide to set up Foundry.

Compile and Deploy:

forge create --rpc-url https://sepolia.infura.io/v3/YOUR_INFURA_KEY --private-key YOUR_PRIVATE_KEY src/EcommerceEscrow.sol:EcommerceEscrow

Replace YOUR_INFURA_KEY and YOUR_PRIVATE_KEY with your credentials. This deploys the contract to Sepolia, giving it a unique address.

Why It Matters

The smart contract is your DApp’s trustless backend. It ensures that funds are securely held in escrow and only released or disputed based on predefined rules. The emitted events (OrderCreated, FundsReleased, DisputeRaised) will be indexed in the next step.

Step 2: The Graph (On-Chain Data Indexing)

Smart contracts store data and emit events, but querying this data directly from the blockchain is slow and inefficient, especially for complex data like mappings or historical events. The Graph solves this by indexing on-chain data and exposing it via a GraphQL API.

What You’ll Do

- Create a subgraph to listen for events from your EcommerceEscrow contract.

- Define a schema to structure the indexed data.

- Write mappings to process events and store data.

- Deploy the subgraph to The Graph’s hosted service or a decentralized network.

Sample Code: Subgraph Configuration

Below is an example of a subgraph for the EcommerceEscrow contract.

schema.graphql

type Order @entity {
  id: ID!
  buyer: Bytes!
  seller: Bytes!
  amount: BigInt!
  isReleased: Boolean!
  isDisputed: Boolean!
}


type OrderCreated @entity {
  id: ID!
  orderId: BigInt!
  buyer: Bytes!
  seller: Bytes!
  amount: BigInt!
}

mapping.ts

import { OrderCreated as OrderCreatedEvent } from "../generated/EcommerceEscrow/EcommerceEscrow";
import { Order, OrderCreated } from "../generated/schema";


export function handleOrderCreated(event: OrderCreatedEvent): void {
  let order = new Order(event.params.orderId.toString());
  order.buyer = event.params.buyer;
  order.seller = event.params.seller;
  order.amount = event.params.amount;
  order.isReleased = false;
  order.isDisputed = false;
  order.save();


  let orderCreated = new OrderCreated(event.transaction.hash.toHex() + "-" + event.logIndex.toString());
  orderCreated.orderId = event.params.orderId;
  orderCreated.buyer = event.params.buyer;
  orderCreated.seller = event.params.seller;
  orderCreated.amount = event.params.amount;
  orderCreated.save();
}

subgraph.yaml

specVersion: 0.0.5
schema:
  file: ./schema.graphql
dataSources:
  - kind: ethereum
    name: EcommerceEscrow
    network: sepolia
    source:
      address: "YOUR_CONTRACT_ADDRESS"
      abi: EcommerceEscrow
      startBlock: YOUR_START_BLOCK
    mapping:
      kind: ethereum/events
      apiVersion: 0.0.7
      language: wasm/assemblyscript
      entities:
        - Order
        - OrderCreated
      abis:
        - name: EcommerceEscrow
          file: ./abis/EcommerceEscrow.json
      eventHandlers:
        - event: OrderCreated(uint256,address,address,uint256)
          handler: handleOrderCreated
      file: ./src/mapping.ts

Deploying the Subgraph

Install The Graph CLI:

npm install -g @graphprotocol/graph-cli

Authenticate and Deploy:

graph auth --studio YOUR_ACCESS_TOKEN
graph deploy --studio YOUR_SUBGRAPH_NAME

Replace YOUR_CONTRACT_ADDRESS, YOUR_START_BLOCK, YOUR_ACCESS_TOKEN, and YOUR_SUBGRAPH_NAME with your details. This deploys the subgraph, giving you a GraphQL endpoint (e.g., https://api.thegraph.com/subgraphs/name/YOUR_SUBGRAPH_NAME).

Querying the Subgraph

You can query the subgraph using GraphQL. For example:

query {
 orders {
  id
  buyer
  seller
  amount
  isReleased
  isDisputed
 }
}

Why It Matters

The Graph makes it easy to query on-chain data like order details or transaction histories without directly interacting with the blockchain, which is slow and costly. The GraphQL API simplifies frontend development by providing structured data.

Step 3: Off-Chain Data Layer (Optional)

Blockchains are not suited for storing large or dynamic data like product images or descriptions due to high gas costs. Instead, you can use off-chain storage solutions and reference them on-chain.

What You’ll Do

- Store large data (e.g., product images, descriptions) on IPFS.

- Optionally, use a Web2 database like Sup Supabase for faster queries, authentication, or analytics.

- Store only a reference (e.g., IPFS CID) on the smart contract.

Example: Storing Product Data on IPFS

1. Upload to IPFS: Use a service like Pinata or Infura to upload files to IPFS. For example, upload a product image and description as a JSON file:

{
 "name": "Laptop",
 "description": "High-performance laptop",
 "image": "ipfs://Qm.../laptop.jpg"
}

This generates a CID like Qm....

2. Reference in Smart Contract: Modify the Order struct in EcommerceEscrow.sol to include a metadataCID field:

struct Order {
  uint256 id;
  address buyer;
  address seller;
  uint256 amount;
  bool isReleased;
  bool isDisputed;
  string metadataCID; // IPFS CID
}

Update the createOrder function to accept and store the CID.

3. Using Supabase (Optional): For faster queries or user authentication, set up a Supabase project and store metadata like:

CREATE TABLE products (
 id SERIAL PRIMARY KEY,
 name TEXT,
 description TEXT,
 image_url TEXT,
 ipfs_cid TEXT
);

Why It Matters

Off-chain storage reduces gas costs and improves scalability. By storing only CIDs on-chain, you maintain decentralization while leveraging IPFS for data availability or Web2 databases for performance.

Step 4: Frontend (Web2 + Web3 Hybrid UI)

The frontend is where users interact with your DApp. It connects to the smart contract, subgraph, and off-chain data to provide a seamless experience.

What You’ll Do

- Build a frontend using Vite and React.

- Use viem to interact with the smart contract.

- Use graphql-request to query the subgraph.

- Use axios to fetch off-chain data from IPFS or Supabase.

Sample Code: React Frontend

Below is a simple React frontend using Vite to interact with the EcommerceEscrow contract and subgraph.

import React, { useState, useEffect } from 'react';
import { createClient } from 'viem';
import { http, createWalletClient, custom } from 'viem';
import { sepolia } from 'viem/chains';
import { request } from 'graphql-request';
import axios from 'axios';
import ecommerceEscrowABI from './abis/EcommerceEscrow.json';

const client = createClient({
  chain: sepolia,
  transport: http('https://sepolia.infura.io/v3/YOUR_INFURA_KEY'),
});

const walletClient = createWalletClient({
  chain: sepolia,
  transport: custom(window.ethereum),
});

const contractAddress = 'YOUR_CONTRACT_ADDRESS';
const subgraphUrl = 'YOUR_SUBGRAPH_URL';

const App = () => {
  const [account, setAccount] = useState(null);
  const [orders, setOrders] = useState([]);
  const [seller, setSeller] = useState('');
  const [amount, setAmount] = useState('');
  const [metadataCID, setMetadataCID] = useState('');

  // Connect to wallet
  const connectWallet = async () => {
    const [address] = await walletClient.requestAddresses();
    setAccount(address);
  };

  // Create order
  const createOrder = async () => {
    const { request } = await client.simulateContract({
      address: contractAddress,
      abi: ecommerceEscrowABI,
      functionName: 'createOrder',
      args: [seller, metadataCID],
      value: BigInt(amount * 10 ** 18), // Convert to wei
      account,
    });
    await walletClient.writeContract(request);
    alert('Order created!');
  };

  // Fetch orders from subgraph
  const fetchOrders = async () => {
    const query = `
      {
        orders {
          id
          buyer
          seller
          amount
          isReleased
          isDisputed
        }
      }
    `;
    const response = await request(subgraphUrl, query);
    setOrders(response.orders);
  };

  // Fetch metadata from IPFS
  const fetchMetadata = async (cid) => {
    const response = await axios.get(`https://ipfs.io/ipfs/${cid}`);
    return response.data;
  };

  useEffect(() => {
    fetchOrders();
  }, []);

  return (
    <div className="p-4">
      <h1 className="text-2xl font-bold">Ecommerce Escrow DApp</h1>
      {!account ? (
        <button onClick={connectWallet} className="bg-blue-500 text-white p-2 rounded">
          Connect Wallet
        </button>
      ) : (
        <div>
          <p>Connected: {account}</p>
          <div className="my-4">
            <input
              type="text"
              placeholder="Seller Address"
              value={seller}
              onChange={(e) => setSeller(e.target.value)}
              className="border p-2 mr-2"
            />
            <input
              type="number"
              placeholder="Amount (ETH)"
              value={amount}
              onChange={(e) => setAmount(e.target.value)}
              className="border p-2 mr-2"
            />
            <input
              type="text"
              placeholder="IPFS CID"
              value={metadataCID}
              onChange={(e) => setMetadataCID(e.target.value)}
              className="border p-2 mr-2"
            />
            <button onClick={createOrder} className="bg-green-500 text-white p-2 rounded">
              Create Order
            </button>
          </div>
          <h2 className="text-xl font-semibold">Orders</h2>
          <ul>
            {orders.map((order) => (
              <li key={order.id} className="my-2">
                Order {order.id}: {order.amount / 10 ** 18} ETH from {order.buyer} to {order.seller}
              </li>
            ))}
          </ul>
        </div>
      )}
    </div>
  );
};

export default App;

Why It Matters

The frontend ties together the smart contract, subgraph, and off-chain data. Users can connect their wallets (e.g., MetaMask), create orders, and view order details fetched from the subgraph. Libraries like viem and graphql-request simplify blockchain and data interactions.

Putting It All Together

Here’s how the components work together in the e-commerce escrow DApp:

1. A user connects their wallet and enters a seller’s address, payment amount, and IPFS CID via the frontend.

2. The frontend uses viem to send a transaction to the EcommerceEscrow smart contract, calling createOrder and emitting an OrderCreated event.

3. The subgraph indexes the OrderCreated event and updates the GraphQL API with the new order data.

4. The frontend queries the subgraph using graphql-request to display the updated order list.

5. If the order includes an IPFS CID, the frontend uses axios to fetch metadata (e.g., product details) from IPFS and displays it.

Conclusion

Building a Web3 DApp involves orchestrating smart contracts for on-chain logic, The Graph for data indexing, IPFS or Web2 databases for off-chain storage, and a hybrid frontend for user interaction. By following this detailed workflow and using the provided code samples, you can create a fully functional DApp like an e-commerce escrow system.
